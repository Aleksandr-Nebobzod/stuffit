5. В детальном чате только по вопросам обозначенного контекста. Если ты сочтёшь что вопросы выходят за этот архитектурный модуль, предложи создать новый чат со своим контекстом.
6. После завершения обсуждения в чате — предложи пользователю обновить глобальный контекст, если появились новые решения.

## 🧱 Архитектура (утверждено)

- Платформа: **Kotlin Multiplatform**
- Начальная цель: **Compose Web** (быстрый старт, без возни с iOS и магазинами)
- Бэкенд: **Firebase** (Auth + Firestore) — допускаем, что потом миграция на Ktor
- Для совместной работы с веб-дизайнером: **отдельный Ktor сервер** (мок-бэкенд, не зависит от Compose)
- Без локальной БД в MVP (оффлайн — сознательно отложено)

## 👤 Пользователи (утверждено)

- **Обязательно** email при регистрации (не анонимный вход)
- Парольная аутентификация (Firebase Auth)
- Привязка к учётным записям из сайд-проекта **Attention Schulte Plus** (в перспективе — единый вход)
- Поля пользователя: id, email, displayName?, createdAt, settings

## ⏸ Отложено (сознательно)

- Оффлайн-режим и синхронизация
- Загрузка аватарок
- Роли пользователей
- Пуш-уведомления

## 🔗 Связи между чатами (важно)

- `2-user-environment` даёт `userId`, к которому привязаны все остальные сущности
- Решения из `2-user-environment` (например, "уведомления выключены по умолчанию") влияют на `6-ranking` и `7-ai-advice`
- Решения из `3-item` (например, "добавлен статус active/loaned/sold") влияют на `5-exchange`
- `8-ktor-ui` использует интерфейсы `AuthRepository` из `2-user-environment` (в режиме мок-заглушки)
- **`9-data-schema` является источником истины для всех моделей данных.** Детальные чаты (`2`-`7`) должны синхронизировать свои модели с общей схемой.
